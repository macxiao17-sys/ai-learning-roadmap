## 🗺️ 学习路线

### 第一阶段：Transformer深度理解（4-6周）

#### 目标
- 不只是"会用"Transformer，而是能从数学层面解释每个组件
- 能手写实现一个mini-Transformer

#### 学习内容

**1. Self-Attention机制**

```
Attention(Q, K, V) = softmax(QK^T / √d_k) × V

必须理解的细节：
├── 为什么除以√d_k？（防止softmax饱和）
├── Multi-Head的作用？（不同子空间的表示）
├── Causal Mask的实现？（下三角矩阵）
├── KV Cache的原理？（避免重复计算）
└── 复杂度分析？（O(n²d)）
```

**2. 位置编码**

| 类型 | 原理 | 代表模型 | 优劣 |
|------|------|---------|------|
| 正弦位置编码 | 固定的sin/cos函数 | GPT-2 | 不能泛化到训练长度外 |
| 可学习位置编码 | 可训练的嵌入向量 | GPT-2 | 参数量随长度增加 |
| **RoPE（旋转位置编码）** | 旋转矩阵编码相对位置 | LLaMA/Qwen | ✅ 当前主流 |
| ALiBi | 注意力偏置 | BLOOM | 零样本长度泛化 |

**3. FFN（前馈网络）变体**

```
标准FFN：FFN(x) = W2 × ReLU(W1 × x) + b

现代变体（更高效）：
SwiGLU：FFN(x) = W2 × (Swish(W1 × x) ⊙ W3 × x)
GeGLU：FFN(x) = W2 × (GELU(W1 × x) ⊙ W3 × x)

为什么GLU更好？门控机制能更好地控制信息流
```

**4. 归一化层**

| 类型 | 公式 | 使用模型 |
|------|------|---------|
| LayerNorm | 标准层归一化 | GPT-2/BERT |
| **RMSNorm** | 仅缩放，不中心化 | LLaMA/Qwen/Gemma |
| Pre-Norm | 归一化在Attention之前 | 大多数现代模型 |
| Post-Norm | 归一化在Attention之后 | 原始Transformer |

#### 实操项目
```
项目：从零实现miniGPT
1. 用PyTorch实现一个2层Transformer（~10M参数）
2. 实现RoPE位置编码
3. 实现SwiGLU FFN
4. 在WikiText-2上训练，验证loss下降
5. 能生成简单文本

参考：Karpathy的nanoGPT，但要自己写，不要直接抄
```

**2. 数据清洗Pipeline**

```
原始数据 → 语言识别 → 长度过滤 → 质量过滤 → 去重 → 安全过滤 → 最终数据
```

每一步详解：

| 步骤 | 方法 | 工具 |
|------|------|------|
| 语言识别 | fastText语言分类器 | fasttext |
| 长度过滤 | 按token数/字符数过滤 | 自定义脚本 |
| 质量过滤 | perplexity过滤（KenLM） | KenLM |
| | 分类器过滤（fastText） | fastText |
| | 规则过滤（特殊字符比例） | 自定义 |
| 去重 | MinHash + LSH | datasketch |
| | SimHash | 自定义 |
| | 精确去重（SHA256） | hashlib |
| 安全过滤 | 有毒内容分类器 | perspective-api |
| | PII去除（邮箱/电话） | presidio |

**3. 数据配比**

```
关键发现：数据配比对模型能力影响巨大

典型配比（参考LLaMA/Qwen）：
├── 通用网页：60-70%
├── 代码：10-15%
├── 学术论文：5-10%
├── 书籍：5-10%
├── 百科/知识：5-10%
└── 对话/社交：3-5%

代码比例高的模型，推理能力更强（Code-LLaMA的发现）
```

**4. Tokenizer**

| 算法 | 原理 | 使用模型 |
|------|------|---------|
| BPE（字节对编码） | 迭代合并高频字符对 | GPT系列 |
| WordPiece | 类似BPE，基于似然 | BERT |
| Unigram | 从大词表逐步裁剪 | T5 |
| SentencePiece | 语言无关的分词 | LLaMA |

```
关键决策：
├── 词表大小：32K-100K（大词表→多语言好，小词表→效率高）
├── 特殊Token：[BOS], [EOS], [PAD], [UNK]
├── 中文处理：字切分 vs 词切分
└── 训练数据：用你的语料训练tokenizer（重要！）
```

### 第三阶段：预训练实战（8-12周）

#### 目标
- 能在小规模数据上预训练一个语言模型
- 理解训练超参数的影响
- 掌握Scaling Laws

#### 学习内容

**1. 训练配置详解**

```python
# 典型预训练配置
training_config = {
    # 学习率
    "learning_rate": 3e-4,           # 基础学习率
    "min_lr": 3e-5,                  # 最小学习率（cosine末尾）
    "warmup_steps": 2000,            # 预热步数
    "lr_scheduler": "cosine",        # 余弦退火
    
    # 优化器
    "optimizer": "AdamW",
    "beta1": 0.9,
    "beta2": 0.95,
    "weight_decay": 0.1,
    "grad_clip": 1.0,
    
    # 批次大小
    "batch_size": 4,                 # 每卡batch
    "gradient_accumulation": 8,      # 梯度累积
    "effective_batch": 32,           # 有效batch = 4 × 8 × 1卡
    
    # 精度
    "dtype": "bf16",                 # 优先BF16
    "loss_scaling": "dynamic",
    
    # 其他
    "dropout": 0.0,                  # 大模型通常不用dropout
    "activation_checkpointing": True, # 显存不足时开启
}
```

**2. 权重初始化**

| 方法 | 公式 | 适用 |
|------|------|------|
| GPT-2初始化 | 残差层 × 1/√(2×N) | GPT系列 |
| Xavier | std = √(2/(fan_in+fan_out)) | 通用 |
| 小型正态分布 | std=0.02 | 小模型 |

**3. Scaling Laws**

```
核心发现（来自OpenAI/Kaplan Chinchilla）：

Loss ≈ (C/C_0)^(-α) + L_∞

其中：
- C = 计算预算（FLOPs）
- α ≈ 0.05
- L_∞ = 数据集的理论最低loss

Chinchilla最优（2022）：
├── 参数量 N 和 数据量 D 应该同比例增长
├── 最优比例：D ≈ 20 × N（tokens）
├── 例：7B模型需要 ~140B tokens
└── 例：70B模型需要 ~1.4T tokens

实际中通常"数据过采样"：
├── LLaMA：7B模型用了1T tokens（远超20×N）
└── 原因：推理成本比训练成本更重要
```

**4. MoE（混合专家模型）**

```
核心思想：不是所有参数都参与每次推理

架构：
├── 多个专家网络（每个是FFN）
├── 路由网络（Router）决定每个token用哪些专家
├── Top-K路由：每个token选K个专家
└── 负载均衡：防止所有token都路由到同一个专家

优势：
├── 参数量大但计算量小（稀疏激活）
├── 例：Mixtral 8x7B = 47B参数，但每个token只激活12B
└── 训练效率高，但通信复杂

关键问题：
├── 负载均衡损失（Load Balancing Loss）
├── 专家崩塌（所有token都选同一个专家）
└── 专家间通信（AllToAll）
```

#### 实操项目
```
项目：预训练一个小模型
1. 架构：GPT-2 Small（~125M参数）
2. 数据：自己清洗的中文语料（10-20GB）
3. Tokenizer：自己训练的sentencepiece
4. 训练：单卡4070 SUPER，训练3-5天
5. 监控：loss曲线、学习率曲线、梯度norm
6. 评估：在验证集上评估perplexity
7. 生成：测试模型生成的文本质量
```

### 第四阶段：SFT与对齐（6-8周）

#### 目标
- 掌握监督微调（SFT）的完整流程
- 理解RLHF/DPO/GRPO的原理与实现
- 能让模型遵循指令、安全对齐

#### 学习内容

**1. SFT（Supervised Fine-Tuning）**

```
数据格式（ChatML）：
<|system|>
你是一个有用的AI助手。
<|user|>
什么是机器学习？
<|assistant|>
机器学习是...
```

| 技术 | 原理 | 显存需求 | 适用场景 |
|------|------|---------|---------|
| Full Fine-tuning | 全参数微调 | 高 | 数据充足、算力充足 |
| **LoRA** | 低秩适配 | 低 | ✅ 最常用 |
| **QLoRA** | 量化+LoRA | 极低 | ✅ 消费级GPU |
| Prefix Tuning | 前缀向量 | 低 | 生成任务 |
| Adapter | 插入适配层 | 低 | 分类任务 |

**LoRA详解：**
```
原始权重：W (d × d)
LoRA分解：W = W₀ + BA
├── W₀: 冻结的原始权重
├── B: (d × r) 低秩矩阵
├── A: (r × d) 低秩矩阵
└── r: 秩（通常8-64）

参数量：仅增加 2×d×r（远小于 d²）
```

**2. RLHF（人类反馈强化学习）**

```
完整流程：
Step 1: SFT → 得到SFT模型
Step 2: 收集人类偏好数据 → 训练Reward Model
Step 3: 用PPO优化策略模型

Reward Model：
├── 输入：prompt + response
├── 输出：标量分数
├── 训练：偏好对比（chosen vs rejected）
└── 损失：-log(sigmoid(r_chosen - r_rejected))

PPO算法：
├── 策略梯度：最大化 reward - β × KL散度
├── KL散度约束：防止模型偏离SFT模型太远
├── 需要4个模型：Policy, Reference, Reward, Value
└── 训练不稳定，超参数敏感
```

**3. DPO（Direct Preference Optimization）**

```
核心思想：跳过Reward Model，直接从偏好数据优化策略

数学等价：
RLHF目标 = max E[reward(x,y)] - β KL(π || π_ref)
DPO目标 = max E[-log σ(β (log(π(y_w|x)/π_ref(y_w|x)) - log(π(y_l|x)/π_ref(y_l|x))))]

优势：
├── 不需要单独的Reward Model
├── 训练更稳定
├── 实现更简单
└── 效果接近RLHF

劣势：
├── 对偏好数据质量更敏感
├── 缺乏在线探索
└── 可能过拟合偏好数据
```

**4. GRPO（Group Relative Policy Optimization）**

```
DeepSeek-R1提出的新方法：

核心思想：
├── 不需要Reward Model
├── 不需要Value Model
├── 用规则奖励（Rule-based Reward）
├── 一组采样 → 互相比较 → 相对排名
└── 特别适合数学/代码等有明确答案的任务

奖励设计：
├── 正确性奖励：答案对/错
├── 格式奖励：是否遵循输出格式
├── 语言奖励：是否使用指定语言
└── 惩罚：重复、冗长等

优势：
├── 训练简单，不需要复杂基础设施
├── 数学推理能力强
├── 可扩展性好
└── DeepSeek-R1的核心技术
```

#### 实操项目
```
项目1：LoRA微调
1. 用QLoRA在4070 SUPER上微调Qwen-7B
2. 构造1000条中文指令数据
3. 训练3个epoch，观察loss变化
4. 测试微调前后的效果对比

项目2：DPO对齐
1. 收集1000条偏好对比数据
2. 用DPO训练一个对齐模型
3. 对比SFT vs DPO的效果
4. 分析偏好数据质量对结果的影响
```

## ⚠️ 常见坑与注意事项

1. **数据质量 > 数据数量** — 10GB高质量数据 > 100GB低质量数据
2. **Tokenizer很重要** — 不同Tokenizer对同一文本的token数差异巨大
3. **Loss不降不一定是bug** — 可能是数据问题、学习率问题、数值精度问题
4. **BF16比FP16好** — BF16动态范围大，不容易overflow
5. **小规模实验先行** — 先在小数据+小模型上验证，再扩大规模
6. **评估不能只看loss** — loss低不等于效果好，一定要人工评估
7. **RLHF比SFT难10倍** — 超参数敏感、训练不稳定、需要大量偏好数据
8. **不要忽视数据安全** — 确保训练数据不含敏感信息
